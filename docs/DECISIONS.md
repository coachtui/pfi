# Architecture & Product Decisions

Format: date, decision, context, alternatives, reasoning, consequences. Do not make major structural decisions without recording them here.

---

## 1. 2026-07-15 — Offset-based index instead of naive ratio

**Decision:** `index(v) = 100 + 100 × (v − A) / S` with a median anchor and a floored scale (see FINANCIAL_INDEX_METHODOLOGY.md).
**Alternatives:** naive `current/start × 100`; log-scaled index; percentile-based index.
**Reasoning:** naive ratio explodes for negative/near-zero starts; log scale can't handle negatives and is hard to explain; percentiles need cohort data we don't have yet. The offset form degrades exactly to the naive formula in the healthy case and is explainable in one sentence.
**Consequences:** index points are "percent of starting scale", not strictly "percent growth" for users with unusual anchors; anchor method must be surfaced in the UI.

## 2. 2026-07-15 — Single Next.js app, not a monorepo

**Decision:** one app with `src/lib/financial-engine` kept framework-free.
**Alternatives:** pnpm workspace monorepo (`apps/web`, `packages/financial-engine`, …).
**Reasoning:** one consumer today; monorepo plumbing adds cost with no benefit yet. The brief explicitly allows this. Extraction path is preserved by the no-React-imports rule in `lib/financial-engine` and `lib/demo-data`.
**Consequences:** revisit when the Expo app or a second package consumer starts.

## 3. 2026-07-15 — Supabase wiring deferred to Phase 3

**Decision:** env validation scaffolding now (`lib/config/env.ts`, optional Supabase vars); live auth/persistence when real user data arrives.
**Alternatives:** wire Supabase in Phase 0 per the original roadmap.
**Reasoning:** Phases 0–1 run entirely on deterministic demo data; auth against nothing adds surface without value, and no credentials exist in this environment. Security rules for the first persistence work are pre-committed in SECURITY_MODEL.md.
**Consequences:** Phase 3 must land schema + RLS + auth together; env vars flip from `.optional()` to required then.

## 4. 2026-07-15 — Chart data indexed once over full history; ranges are views

**Decision:** anchor derived from full history; 30D/90D/1Y/All only slice the rendered window.
**Alternatives:** re-anchor per range (each range starts at 100).
**Reasoning:** re-anchoring makes the same day show different values in different ranges — confusing and hides long-term drift. One anchor keeps "118.4" meaning one thing.
**Consequences:** short ranges may render far from 100; Y axis auto-zooms to data.

## 5. 2026-07-15 — Honest cash impact in engine, equity-aware display

**Decision:** `Driver.impact` is always the true cash impact; `driverDisplay()` presents investment contributions and full debt payoffs as equity-positive (emerald, "+") so saving is never framed as loss. Routine debt payments display as outflows.
**Alternatives:** flip signs in the engine; show everything as raw cash flow.
**Reasoning:** engine stays truthful for calculations and future AI input; display layer carries the owner-created-equity product principle (mockups show "Investment +$500" in green).
**Consequences:** display semantics live in one tested function (`driverDisplay`); AI narration must use the same semantics.

## 6. 2026-07-15 — Demo "today" is a fixed date

**Decision:** Koa Holdings history ends at a hard-coded 2026-07-15 with a fixed PRNG seed.
**Alternatives:** generate relative to the real current date.
**Reasoning:** full determinism — identical charts in tests, screenshots, reviews; no snapshot drift between runs.
**Consequences:** the demo dashboard always shows the same "today"; acceptable until real data phases.

## 7. 2026-07-15 — Supabase infrastructure pulled forward ahead of remaining Phase 1 screens

**Decision:** build auth (magic link/PKCE), schema + RLS (migrations 0001/0002), the snapshot builder, demo-seed pipeline, and a DB-backed dashboard now, before the rankings/data/report screens land.
**Alternatives:** finish all Phase 1 screens on demo-only (in-memory) data first and defer persistence to Phase 3 as originally sequenced (see #3).
**Reasoning:** user decision. Validating the real data path — auth, RLS, snapshot derivation — early de-risks the highest-uncertainty parts of the system before more UI is built on assumptions that only hold for demo data. This supersedes #3's deferral: #3's noted consequence ("env vars flip from optional to required then") has now happened.
**Consequences:** ROADMAP's Phase 3 boundary narrows — schema, RLS, and auth are already done, so Phase 3 is now "manual data + CSV import" on top of live infrastructure. Rankings/data/report screens still ship on mock/demo data next, but now sit on top of a real Supabase project rather than a purely synthetic one.

## 8. 2026-07-15 — Daily snapshots store raw dollar components; index computed at read time

**Decision:** `daily_snapshots` (migration 0001) persists raw numeric components — `liquid_assets`, `revolving_balances`, `near_term_obligations`, `essential_obligations`, `safety_buffer`, `net_worth` — not a precomputed index/baseline/waterline.
**Alternatives:** store the computed index, baseline, and waterline per day alongside the components.
**Reasoning:** the index anchor (FINANCIAL_INDEX_METHODOLOGY) is a pure function of the raw component history. Storing only components lets methodology changes (anchor formula, scale floor, future score versions) replay against history without a data migration, and keeps `financial-engine` the single source of truth for derived math.
**Consequences:** every dashboard read recomputes index/baseline/waterline from the stored raw series (cheap — pure functions over a bounded window). `health_score`/`score_version` columns from the original DATA_MODEL draft are deferred to Phase 2 when score work begins; the draft's inline `financial_index`/`available_position` fields are superseded by the implemented raw-components shape.

## 9. 2026-07-15 — Magic-link-only auth

**Decision:** Supabase auth configured for email magic link (PKCE flow) only — no password, no OAuth providers, in this phase.
**Alternatives:** password auth; social OAuth (Google, etc.).
**Reasoning:** magic link removes password storage/reset surface entirely, matching a personal-finance product's low-friction, low-attack-surface bar for this phase. PKCE via `/auth/callback` is the modern secure pattern for browser redirect auth.
**Consequences:** `admin.generateLink` (used for local dev bootstrap) emits implicit tokens, not PKCE codes, so `scripts/dev-login.ts` can't hit the code-only callback directly — it uses `verifyOtp` instead. The production email-click flow is architecturally sound but unverified end-to-end with a real inbox (see KNOWN_LIMITATIONS). All future auth settings must live in `supabase/config.toml`: `supabase config push` syncs the whole `[auth]` section, so any setting configured only via dashboard is reverted on the next push (see SECURITY_MODEL).

## 10. 2026-07-15 — Demo data seeds through the real persistence pipeline

**Decision:** the demo generator's accounts and transactions are written to Supabase (`financial_accounts`/`transactions` with `provider = 'demo'`) through the same server actions and snapshot builder a real data source would use, rather than a synthetic in-memory-only demo path.
**Alternatives:** keep demo data client-side/in-memory (as in Phase 0–1) and wire real persistence separately later.
**Reasoning:** exercising the real insert → snapshot-build → RLS-read path with demo data is the cheapest way to prove the persistence and security layers work before any real user data exists, and it means the dashboard has no separate demo-only code branch to maintain.
**Consequences:** `clearDemoData` currently deletes by `user_id` (all `financial_events`/`daily_snapshots` for that user, plus `financial_accounts` where `provider = 'demo'`) — correct while demo is the only data source, but must become source-scoped once manual/CSV data coexists with demo data (see KNOWN_LIMITATIONS).

## 11. 2026-07-15 — `middleware.ts` renamed to `proxy.ts`

**Decision:** the Next.js route guard lives at `src/proxy.ts`, not `src/middleware.ts`.
**Alternatives:** keep the `middleware.ts` name used through Phase 0–1 planning and in the original ARCHITECTURE.md wording.
**Reasoning:** Next.js 16 renamed the route-interception convention from `middleware` to `proxy`; the app pins to Next 16.2.10, so the new filename is required for Next to pick it up.
**Consequences:** "middleware" in older docs/discussion refers to what is now `src/proxy.ts`; re-check this convention name on future Next major-version upgrades.

## 12. 2026-07-15 — Transactions/categorization UI will be a dashboard drill-down, not a nav item

**Decision:** when Phase 3 ships the transaction list and correction workflow (recategorization, essential/discretionary marking, transfers, splits), it will live as a drill-down from the Home dashboard (e.g. tapping "Available Capital" or "What moved your line" → transaction detail), not as a fifth bottom-nav tab or a settings subsection. User approved this direction.
**Alternatives:** fifth nav item ("Accounts"); accounts section behind profile/settings.
**Reasoning:** the mockups define a four-tab nav (Home/Rankings/Data/Report) with no transactions surface, and the product addendum's UX hierarchy requires the diagnosis (score, drivers, actions) above dense transaction lists — a drill-down keeps detail beneath the diagnosis instead of promoting it to a peer destination.
**Consequences:** Phase 3's plan should design the drill-down entry points and routes (likely `/accounts` or `/transactions` reachable from dashboard cards); bottom nav stays four tabs; deep-linking to a specific transaction remains possible via route, just not via nav.

## 13. 2026-07-16 — Uniform override edit model; archive-not-delete accounts; manual txns only on manual accounts

**Decision:** all transaction edits — demo or manual — write `user_override` (category/description) or the mutable `notes` column; amount/date mistakes on manual transactions are fixed by delete + re-add. Accounts are archived (`archived_at`), never deleted. Manual transactions may only be created in `provider='manual'` accounts; imported (demo/csv) transactions can never be deleted. Snapshot rebuilds read source columns only, so overrides never move the index.
**Alternatives:** relaxing the 0002 immutability trigger for manual rows; hard-deleting accounts; allowing manual txns in demo accounts.
**Reasoning:** one provenance rule for every provider keeps the trigger untouched and the audit trail complete; account.provider stays the single source of a transaction's origin; archived accounts preserve history for past snapshots.
**Consequences:** recategorizing to/from `income` changes report groupings but not obligations or the index (v1 — revisit with the Phase 2 metric registry); manual edit UX for amount/date is delete + re-add.

## 14. 2026-07-16 — PFI Score v1: six weighted dimensions, unscored Protection, Momentum overlay, compute-at-read

**Decision:** the v1 health score uses six weighted dimensions (Cash Flow 25 / Liquidity 20 / Debt 20 / Stability 15 / Growth 15 / Concentration 5). Protection is displayed as a status (`not_assessed` in v1) and carries no weight; Momentum is a deterministic directional overlay derived from score history, not a seventh dimension. Missing data follows a deterministic policy: dimensions without their required core metric are ineligible (never neutral-filled), weights renormalize over eligible dimensions, a full score requires Cash Flow + Liquidity eligible and ≥4/6 eligible, 4–5 eligible is labeled provisional, otherwise the overall score is suppressed. Scores are computed at read time as a pure function of snapshots/transactions/accounts + `PFI_SCORE_VERSION` — no `health_score` columns, no migration. Score deltas follow the shared 30D/90D/1Y/All range picker. User-facing copy uses consumer language (monthly surplus, emergency runway, debt burden — no FCF/owner-created-equity jargon in score UI). FINANCIAL_HEALTH_SCORE.md is the normative spec.
**Alternatives:** seven weighted dimensions with Protection at 5% (prior provisional spec); neutral-filling or range-capping missing dimensions; Momentum as a weighted dimension; persisting daily scores in `daily_snapshots` (migration 0004); a fixed 30-day delta period.
**Reasoning:** Protection has no direct data source in v1 and inferring it from bank data would fabricate a number; Momentum as a dimension would double-count movement already captured per dimension; eligibility + renormalization is the only missing-data strategy that neither invents nor punishes; compute-at-read matches DECISIONS #8 (derived values at read time) and rebuildSnapshots() rewrites snapshot rows on every mutation anyway, so stored scores would not actually preserve history today.
**Consequences:** version bumps recompute displayed history (disclosed in UI) until scores are persisted — revisit persistence when non-rebuildable provider data lands (Phase 7); adding Protection to the weighted score later requires documented methodology, direct user inputs, missing-data rules, compliance review, and a version bump; the report screen's "Free cash flow" / "Owner-created equity" labels need consumer-language relabeling for consistency.

## 15. 2026-07-17 — CSV import architecture

**Decision:** client-side parse (the raw file never leaves the browser) feeding a server-action trust boundary that re-validates and re-derives everything server-side; a framework-free `src/lib/csv-import/` module (parse/detect/normalize/dedupe/transfers); provenance via `transactions.import_batch_id` only — no staging table, no `import_batches` table, batch summaries are derived by grouping on `import_batch_id` at read time (`getRecentImports`); imported rows are corrected via `user_override` (same uniform edit model as DECISIONS #13) and removed only by whole-batch undo (`undoImport`), never a per-row delete; transfer pairs are recorded on the new row only, since migration 0002's source-column immutability trigger (extended in migration 0004 to cover `import_batch_id`) forbids updating an existing row's `transfer_pair_id` to point at a newly imported counterpart.
**Alternatives:** server-side parse + staging table (rejected — more moving parts, no v1 benefit: the browser is already a trusted-enough boundary for a file the user chose to upload, and a staging table just adds a second write path to keep consistent with the real `transactions` table); one-shot import without preview (rejected — guts the review loop that lets a user catch a wrong column mapping or sign convention before anything is committed).
**Reasoning:** matches DECISIONS #8's compute/derive-at-read-time pattern (no new persisted-summary table to keep in sync) and #13's uniform override model (one correction mechanism for every transaction provider, not a CSV-specific one); keeping `csv-import/` framework-free preserves the same future-extraction path already committed to for `financial-engine`/`demo-data`. See `docs/superpowers/specs/2026-07-17-csv-import-design.md` for full rationale, the UI flow, and the out-of-scope list.
**Consequences:** an import batch has no first-class row of its own — undo and "recent imports" are always a derived query over `transactions.import_batch_id`, so deleting the last row of a batch makes that batch disappear from history with no separate record; when the counterpart of a transfer pair already exists as a prior transaction, that existing row cannot be updated to carry the pair link, so its confidence penalty for being an unresolved transfer persists even after the new row imports as the matched half (documented in KNOWN_LIMITATIONS).

## 16. 2026-07-17 — Drop the write-only `daily_snapshots.data_coverage_confidence` column

**Decision:** migration `0005_drop_coverage_confidence` removes the column; `snapshotToRow` stops stamping it.
**Alternatives:** stamp it meaningfully per rebuild (derive demo/manual/mixed from the providers of accounts feeding that rebuild); keep it and doc-note it as vestigial until Phase 7.
**Reasoning:** the column was write-only — stamped `"demo"` unconditionally, never read. Score confidence is computed at read time from account providers (#14, `metric-inputs.ts`), which made the persisted field obsolete; after the manual-CRUD and CSV-import slices it actively misstated provenance for rebuilt manual/mixed snapshots. A field that lies is worse than no field, and speculative stamping logic nothing consumes fails YAGNI.
**Consequences:** schema stays honest. If persisted coverage confidence returns with real provider sync (Phase 7), it gets a fresh design with real inputs (sync freshness, connection health) rather than inheriting a placeholder.

## 17. 2026-07-17 — Demo profile registry with data-only switching

**Decision:** three hand-authored deterministic demo profiles (Koa Holdings unchanged; Blue Reef Partners — early-career under strain; North Shore Capital — debt-free, concentrated) behind a metadata registry (`src/lib/demo-data/profiles.ts`, client-safe) and a server-side generator map (`generators.ts`); `loadDemoData(profileId)` validates against the registry and defaults to Koa; switching replaces demo rows only (provider-scoped clear → seed → rebuild) and never touches the user's company identity or cohorts; the active profile is detected at read time by matching seeded demo-account display names against per-profile signature names — no schema change.
**Alternatives:** one parameterized generator (rejected — rewrites the tuned, test-pinned Koa generator for no user-visible gain); full persona switch overwriting `personal_companies`/`user_profiles` (rejected — destructive to the user's own onboarding identity, needs save/restore bookkeeping); onboarding-only profile choice (rejected — reviewers couldn't flip personas in one session); persisting the active profile id in a column (rejected — a schema change for derivable demo-only state).
**Reasoning:** the two new personas exist to exercise states Koa can't (below-waterline, low bands, high utilization, irregular income; debt-free rule, concentration penalty, high bands), and per-profile persona-invariant tests prove they actually do; splitting metadata from generators keeps generator code out of client bundles; signature-name detection is an accepted heuristic for demo-only state (KNOWN_LIMITATIONS).
**Consequences:** demo-data UI copy must keep the profiles clearly fictional; renaming a signature account in a generator without updating its registry meta breaks detection (guarded by a registry test); the switcher gives `clearDemoData` its first UI entry point.

## 18. 2026-07-17 — Paginate `rebuildSnapshots`' unbounded queries past PostgREST's 1000-row cap

**Decision:** `src/lib/data/rebuild-snapshots.ts`'s `transactions` and `daily_snapshots` (prior) fetches now go through a new `paginateAll` helper (`src/lib/data/paginate.ts`) that pages in batches of 1000 via `.range()` until a short page confirms the end, instead of a single unbounded `.select()`.
**Alternatives:** raise PostgREST's `db-max-rows` config (rejected — a project-wide config change with broader blast radius, doesn't fix any other unbounded query, and doesn't help until every environment is reconfigured); leave it and only document it (rejected — this silently produces wrong obligations/score numbers for real users, not just degraded accuracy, so document-only under-serves the actual risk).
**Reasoning:** found live while QA-verifying the demo-profiles slice: Blue Reef Partners' 1042 transactions were silently truncated to 1000 rows by Supabase's default PostgREST cap, which cut off income-date detection after 2026-06-05 (of a 2026-07-15 dataset) and left `near_term_obligations`/`essential_obligations` at exactly 0 for every day past that point — verified directly against the live database and confirmed fixed (recomputed value 150, matching the dataset's true value) after applying `paginateAll`. This is the same class of pre-existing risk KNOWN_LIMITATIONS already flagged for `getImportContext`/`importTransactions` (see the CSV import v1 section), but rebuild-snapshots.ts's instance was worse: it silently corrupts every displayed obligation/score number for the affected date range with no error surfaced, not merely degraded dedupe accuracy on one import.
**Consequences:** `getImportContext`/`importTransactions`'s equivalent unbounded fetches remain unfixed (out of scope for this narrow hotfix) — their KNOWN_LIMITATIONS entry is updated to drop the now-stale "mirrors rebuild-snapshots.ts" framing, since that instance no longer applies. Any account whose transaction or snapshot count grows past 1000 continues to need a `paginateAll` pass at each new unbounded query site; there is no repo-wide guard preventing a future unbounded `.select()` from reintroducing this risk elsewhere.

## 19. 2026-07-17 — Paginate `getImportContext`/`importTransactions`' unbounded queries too

**Decision:** the two remaining unbounded fetches flagged by DECISIONS #18's Consequences — `getImportContext`'s existing-transactions read (`src/lib/data/queries.ts`) and `importTransactions`'s server-side dedupe re-check (`src/app/actions/imports.ts`) — now use the same `paginateAll` helper, ordered by `transactions.id` (unique) for stable pagination.
**Alternatives:** none newly considered — this reuses #18's already-decided approach and helper; the only question was whether to fix it now or later, and the user asked for it directly as a fast follow-up.
**Reasoning:** live-verified the same failure mode: a test account seeded with 1050 transactions returned exactly 1000 rows from the old unbounded query and all 1050 from the paginated one. For `getImportContext` this means the `/import` preview's dedupe/transfer-pair detection was blind to any existing transaction past the 1000th row (by `id` order) — a re-imported file could silently duplicate rows the truncated fetch never saw. For `importTransactions`' dedupe re-check, the same truncation meant the server-side re-validation (the actual trust boundary, since the client's own dedupe output is only advisory) could miss duplicates past the cap and commit them anyway.
**Consequences:** all three of this project's known unbounded-`.select()` sites on `transactions` are now paginated (`rebuild-snapshots.ts`, `getImportContext`, `importTransactions`). `getRecentImports` (`src/lib/data/queries.ts`) has the same un-paginated pattern, scoped to `import_batch_id IS NOT NULL` rows only — not fixed here (out of the explicitly requested scope), added to KNOWN_LIMITATIONS as a known, smaller-blast-radius instance of the same risk.

## 20. 2026-07-17 — Paginate `getRecentImports` too

**Decision:** `getRecentImports` (`src/lib/data/queries.ts`) now uses the same `paginateAll` helper, ordered by `transactions.id`, sharing the `TRANSACTIONS_PAGE_SIZE` constant with `getImportContext` (renamed from `IMPORT_CONTEXT_PAGE_SIZE` now that two functions use it).
**Alternatives:** none newly considered — same helper, same pattern, user asked for this specific site as the next fast follow-up to #19.
**Reasoning:** live-verified with a seeded 1050-transaction account split across two import batches (500 + 550 rows): the old (un-ordered) unbounded query returned exactly 1000 of the 1050 rows and undercounted one batch as 450 (true count 500), while the paginated version correctly returned all 1050 rows and reported 500/550 — proving both the truncation and the fix, though the *old* query's lack of any `.order()` means Postgres's default row order (not a rigorous id/insertion-order explanation) determined exactly which 50 rows were dropped. Either way, a real user would have seen a wrong row count (and potentially wrong first/last dates) on the `/accounts` "Recent imports" list for any batch whose rows fell past the cap.
**Consequences:** all four unbounded-`.select()` sites on `transactions` found via `getImportContext`/`importTransactions`/`getRecentImports`/`rebuild-snapshots.ts` are now paginated. Further audit found the same pattern in `getDashboardData`, `getReportData`, and — the highest-impact of the three — `fetchScoreSources` (feeds `getScoreData`/`getScoreSummary`, meaning the PFI Score itself would silently undercount past 1000 transactions). None of these three are fixed here; added to KNOWN_LIMITATIONS as the next candidates. There is still no repo-wide guard (lint rule or wrapper convention) preventing a future unbounded `.select()` from reintroducing this class of bug.

## 21. 2026-07-17 — Full pagination audit: every growable-table select paginated, one shared convention

**Decision:** after three piecemeal pagination hotfixes (#18–#20) each surfaced another instance, a full audit of every `.from(...).select` in `src/` and `scripts/` classified all sites by whether their table can exceed PostgREST's 1000-row cap per user. Every select on the three growable tables (`transactions`, `daily_snapshots`, `financial_events`) now goes through a new `paginateSelect` helper (`src/lib/data/paginate.ts`, wraps `paginateAll` for Supabase's `{ data, error }` shape) with a stable, unique `.order()` — this pass fixed `fetchScoreSources` (feeds the PFI Score), `getReportData`, `getDashboardData`, and `getTransactionsData`, and converted the four previously-fixed sites to the same helper so exactly one pattern exists. Where the sort key alone isn't unique (`posted_date`, event `date`, `created_at`), `.order("id")` is appended as a tiebreaker, since `.range()` pages over non-unique orderings can skip or duplicate rows.
**Alternatives:** raise PostgREST's `db-max-rows` (rejected in #18 — config-wide blast radius, environment-dependent); an ESLint rule banning un-`range()`d selects (deferred — worth doing, but a custom rule is its own project; the audit + single-helper convention + this record is the interim guard).
**Reasoning:** live-verified all four fixed legs against a seeded account with 1050 transactions, 1100 snapshots, and 1050 events: every old query returned exactly 1000 rows; every paginated query returned the true count with zero duplicated/skipped rows (unique-id count equal to total). Before this fix, a user past 1000 transactions had a silently miscomputed PFI Score (`fetchScoreSources`), wrong report reconciliation (`getReportData`), and a truncated transaction list (`getTransactionsData`); past ~2.7 years of daily snapshots, the dashboard chart itself would have truncated (`getDashboardData`).
**Consequences:** `financial_accounts` selects stay unpaginated by explicit judgment (household scale — documented in KNOWN_LIMITATIONS with the residual risks: that judgment itself, and no automated lint guard). Multi-page fetches add one round-trip per extra 1000 rows, sequential per query but parallel across queries — negligible at household scale. `undoImport`'s delete-RETURNING cap is noted as harmless (only `length === 0` is consumed).

## 22. 2026-07-17 — PWA manifest without a service worker; e2e auth through the real hash flow

**Decision:** installability ships as manifest + icons only (`src/app/manifest.ts` via Next's file convention, all naming from `branding.ts`; committed monogram PNGs with an SVG source and documented regeneration, no native image deps). The Playwright smoke test authenticates by visiting a real `/login#access_token=…` URL — `LoginForm` now establishes a session from implicit-flow hash tokens via `setSession` — with the token minted server-side in global setup (admin `generateLink` → GoTrue `verify` REST) because GoTrue's redirect allowlist is pinned to localhost:3000. `pnpm test:e2e` stays out of `pnpm check` (live-DB, slow), matching `test:rls`/`test:live`.
**Alternatives:** a service worker for offline support (rejected this slice — offline caching of financial data is a privacy design question that must not be decided as a side effect; installability no longer requires a SW); codifying the hand-written-cookie workaround as a Playwright fixture (rejected — leaves the login surface untested and the KNOWN_LIMITATIONS gap open); CI wiring for e2e (deferred — no CI pipeline exists yet).
**Reasoning:** this was ROADMAP Phase 1's last item, and the e2e-auth choice permanently retires the hand-bootstrapped-cookie QA workaround every live-QA session had to rediscover; the hash-processing fix is the exact candidate KNOWN_LIMITATIONS named. Production PKCE links (`?code=` → `/auth/callback`) are untouched. Building the real Playwright harness — rather than another one-off manual QA pass — is exactly what caught two real product bugs on its first run, both invisible to `pnpm check`/`pnpm test:rls` and to every prior manual/screenshot-based QA session: (1) `src/proxy.ts`'s auth matcher redirected unauthenticated `/manifest.webmanifest` requests to `/login`, which would have silently broken PWA installability for every real, logged-out visitor — fixed by adding `manifest.webmanifest` to the matcher's exclusion list, the same pattern already used for `favicon.ico`; (2) `LoginForm.tsx` called `router.refresh()` immediately after `router.replace("/")` in the same tick, which made Next 16's App Router silently drop the server's pending `NEXT_REDIRECT` to `/onboarding` — every real magic-link sign-in got stuck on "Signing you in…" forever, confirmed via direct reproduction before the fix. Both were live for the entire prior Phase 1 slice; neither surfaced in unit tests, RLS checks, or manual QA because manual QA always used the hand-bootstrapped-cookie workaround (KNOWN_LIMITATIONS), which never drives the real `/login` form/redirect path this harness now exercises.
**Consequences:** Phase 1 is complete. E2e runs only locally until CI exists; the smoke suite is the natural place to grow coverage (import flow, profile switching). Offline/SW support needs its own design pass before any financial data is ever cached. The two bugs above were live in production-shaped code for the whole prior slice with no automated signal — the strongest argument yet for prioritizing e2e coverage of auth/redirect paths specifically, since they sit outside what unit tests and RLS checks can see.

## 23. 2026-07-18 — Recurring detection: pure-function engine + override-only persistence

**Decision:** recurring transaction series are detected by a pure, deterministic engine function (`src/lib/financial-engine/recurring.ts`) recomputed from transactions on every snapshot rebuild — nothing detected is persisted. Only user intent is stored (`recurring_overrides`: confirm/dismiss keyed by a stable series hash of account + direction + normalized description, deliberately excluding cadence/amount so overrides survive reclassification). The snapshot builder's obligation windows now split at the edge of known history: actual transactions up to `endDate`, projected occurrences of active non-dismissed series beyond it; the 28-day previous-cycle shift survives only as a fallback when nothing is detected. Recurring confirm/dismiss deliberately moves the index (it curates the obligations projection) — unlike transaction category overrides, which still never do (DECISIONS #13). The unused `transactions.recurring_status` column (0001) was dropped in migration 0006, which also recreated the source-immutability trigger without it.

**Alternatives:** a persisted `recurring_items` table refreshed on import/mutation (stable IDs, but a second source of truth with a staleness pipeline — the same sync class the snapshot rebuild already manages); hybrid persistence of confirmed items only (one concept in two representations); per-transaction `recurring_status` tagging (the 0001 column's original intent).

**Reasoning:** matches "deterministic code calculates"; recompute-on-rebuild is cheap at household volume; the override pattern mirrors `user_override`; detection improves as data arrives with no migration.

**Consequences:** series keys must stay stable across normalizer changes (a normalizer change orphans existing overrides — treat as a versioned methodology change); detection runs on every rebuild and dashboard/accounts load; v1 limits recorded in KNOWN_LIMITATIONS.

## 24. 2026-07-18 — Statement balance anchoring: append-only anchors + roll-forward, snapshot builder untouched

**Decision:** account balances are anchored by append-only `balance_anchors` rows — (date, balance) truth points from statement ending balances entered at CSV import (source `import`, reconciled server-side with the discrepancy recorded) or from manual balance entry (source `manual`, dated that day; account creation and balance-changing edits write one automatically). The engine trusts the *effective anchor* (greatest `anchor_date`, tiebreak `created_at`); `rebuildSnapshots` corrects `financial_accounts.current_balance` to the effective anchor rolled forward through post-anchor transactions on every mutation, so the snapshot builder itself is untouched — its backward replay from `current_balance` now hangs off a maintained value instead of a stale hand-typed one. Anchor entry at import is encouraged but skippable; reconciliation warns (in dollars) and never blocks. Anchorless accounts keep legacy behavior (hand-typed balance authoritative, no roll-forward). Displayed balances are honestly "as of" the anchor — slightly behind reality, provably correct; a 35-day staleness nudge (dismissible, DB-backed, demo data excluded) prompts the monthly ritual.

**Alternatives:** two columns on `financial_accounts` (single anchor, no history — loses the audit trail and per-import discrepancy record; Plaid would force the table later anyway); a multi-anchor engine pinning history piecewise between anchors (a snapshot-builder rewrite; overkill for monthly anchors); requiring the balance at import (blocks users whose exports lack one).

**Reasoning:** transactions determine changes, never levels — the engine needs a trusted (date, balance) point, and a statement's ending balance is the cleanest one that exists (posted-only, bank-reconciled, exact close date — none of the pending-transaction fuzziness of a typed-in app-screen number). Reconciliation gives gap *detection* for free: a mismatch is mathematical proof of missing transactions. ROADMAP Phase 7 (Plaid) becomes "anchor rows from a new source, daily" on the same machinery.

**Consequences:** an import now moves balances for anchored accounts (upgrading the old "adding transactions reshapes history backward and never changes today's balance" quirk into correct behavior for post-anchor transactions); a statement anchor dated earlier than a newer manual anchor does not supersede it — reconciliation surfaces the disagreement instead (KNOWN_LIMITATIONS); batch undo removes its anchor row; discrepancies and freshness are recorded but do not yet feed score confidence (deferred — versioned methodology change).

## 26. 2026-07-18 — AI interpreter core: provider-agnostic narration service + `ai_narrations` cache

**Decision:** Phase 4's first slice narrates the dashboard's existing deterministic "Performance brief" card via the Vercel AI SDK's `generateObject`, addressed through AI Gateway model strings (`src/lib/config/env.ts`'s `PFI_AI_MODEL`, default `anthropic/claude-haiku-4-5`) rather than a pinned provider SDK. The service is optional-key progressive enhancement: `AI_GATEWAY_API_KEY` absent (or empty string, treated as unset by a Zod preprocess step) disables AI narration everywhere, and the dashboard renders the existing deterministic brief with no structural difference — same card, same heading — only the chip text and body differ ("Calculated" vs. "AI narrative · numbers calculated"). A typed Zod boundary (`src/lib/ai/schemas.ts`) is the only channel the model sees or returns through: `NarrationInput` (`.strict()`) carries derived metrics only — available capital, cushion, baseline/waterline comparisons, momentum, up to 4 drivers by TYPE ENUM ONLY (never `FinancialEvent.label`, which can carry user-entered free text), and the score — and `NarrationOutput` (`.strict()`) is a bounded body string (40–700 chars) plus `referencedDriverIds`, checked post-hoc against the input's known driver ids (`referencesOnlyKnownDrivers`) so the model can't fabricate a driver reference. `generateNarration` (`src/lib/ai/narrator.ts`) returns `null` on every failure path — missing key, provider error, timeout (8s `AbortSignal`), schema violation, invented driver reference — so callers always get a clean fallback signal, never a thrown error to catch. A new `ai_narrations` table (migration `0009_ai_narrations.sql`) caches one row per `(user_id, surface, input_hash)` — the hash is a canonicalized (recursively key-sorted) SHA-256 of the exact `NarrationInput` (`src/lib/ai/hash.ts`), so any change to the underlying metrics naturally invalidates the cache with no explicit invalidation logic — and doubles as an audit log (`input_json`, `output_json`, `model`, `created_at`). `getOrGenerateNarration` (`src/lib/data/narration.ts`) checks the cache, generates and upserts on a miss (the upsert is isolated in its own try/catch so a transient DB write failure never discards a successful, costly generation), and never rejects — it returns `Promise<NarrationResult | null>`, which the dashboard passes straight into React's `use()` inside a `Suspense` boundary (`AIPerformanceBrief`), with the deterministic `PerformanceBrief` as both the Suspense fallback and the `null`-result fallback so there's no visible flash between the loading and AI-unavailable states.

**Alternatives:** a direct Anthropic SDK integration (rejected — hand-pins one provider into the narration surface; the gateway model-string approach keeps provider choice a one-line env change and matches this project's other Vercel-first infrastructure choices); a hand-rolled fetch client against a provider's REST API (rejected — reimplements retries, schema-validated structured output, and timeout handling the AI SDK already provides, for no control benefit at this scale); generating narration at snapshot-rebuild time instead of on-demand at dashboard read (rejected — rebuilds run on every mutation, which would call the model far more often than a user actually views the dashboard, and would plumb AI calls into the same synchronous, non-transactional rebuild path KNOWN_LIMITATIONS already flags as fragile enough without adding more work to it); ephemeral/no-persistence narration, calling the model fresh on every load (rejected — the same input produces the same verified metrics, so an unpersisted design would pay the latency and cost of a fresh generation on every dashboard visit for no user-visible benefit, and would forfeit the audit trail this table gives Phase 4 for free).

**Reasoning:** the binding product rule ("deterministic code calculates; AI only narrates," `docs/AI_RECOMMENDATION_POLICY.md`) is enforced structurally, not just by convention — the Zod boundary is the only channel between the engine and the model in either direction, `.strict()` makes smuggling an extra field a runtime error, and the driver-reference check makes the AI unable to talk about anything the deterministic layer didn't already compute. Excluding `FinancialEvent.label` from the boundary (drivers carry `kind: FinancialEventType` only) closes the one path by which user-entered free text could reach a third-party model — labels are exactly the kind of user-typed content (a manually typed transaction description, a note) the SECURITY_MODEL's "masked identifiers only" / analytics-privacy rules already treat as sensitive. Caching by input hash rather than by wall-clock time or a TTL means the cache is exactly as fresh as the underlying data — a cushion or driver set changing invalidates it automatically, and an unchanged dashboard never re-pays generation cost.

**Consequences:** `ai_narrations` rows accumulate one per distinct input hash with no pruning yet (KNOWN_LIMITATIONS) — a household whose numbers change often will accrue one row per distinct day's metrics; narration is single-shot with no streaming — acceptable for a short paragraph, worth revisiting if Phase 4's later surfaces (weekly brief, quarterly report) run long enough to want progressive rendering; the 4-driver cap and 700-character output bound mean the brief only ever covers a small window's most significant events, matching the existing deterministic brief's scope. **Live QA against a real `anthropic/claude-haiku-4-5` model (post-merge-review, before merge) found two bugs no keyless/mocked-model test could see:** `bodyOnlyReferencesKnownAmounts` initially rejected valid narrations that summarized two same-direction drivers as a total (e.g. "$6,900" for two $3,450 paychecks) — correct arithmetic, not a hallucination — fixed by extending the known-amount set with total inflows/outflows/net driver impact (KNOWN_LIMITATIONS records the narrower residual: an aggregate outside that set still falls back safely, just more often than necessary); and the model wrote internal driver ids directly into user-facing prose despite the schema's separate `referencedDriverIds` field existing for that purpose, fixed via a prompt reword. Both fixes were verified against the real model and the full live flow (cold generation, cache-hit-on-reload, the disclosure panel, no-key fallback with a cached row present) at 390×844 and 1280×900 with zero console errors.

## 28. 2026-07-19 — Password auth replaces magic-link sign-in

**Decision:** login is now email-or-username + password (`signInWithPassword`, `src/app/actions/auth.ts`). A username identifier is resolved to an email server-side, using the service-role client only (`emailForIdentifier` — `user_profiles` is unreadable pre-auth under RLS, so an anonymous browser can never run this lookup itself), case-insensitively via a new `user_profiles_username_lower_key` unique index (migration `0010_password_auth.sql`, alongside `escapeLikePattern` to keep the `ilike()` lookup an exact match rather than a wildcard scan). Every login failure — unknown email, unknown username, wrong password — returns the identical `INVALID_CREDENTIALS` message, never revealing which case applied. New surfaces: `/signup` (email + password + a required consent checkbox), `/auth/reset` + `/auth/reset/update` (request/complete a password reset through the existing `/auth/callback` code-exchange route), and `/consent` (a post-login gate for versioned Terms/Privacy). Consent is recorded in a new append-only `user_agreements` table (`(user_id, document, version)`, no update/delete RLS policies) at sign-up time via a service-role insert (the user has no session yet to write under their own RLS), and re-checked post-login by `src/proxy.ts`, which now also enforces the gate — one DB query per session, cached by a cookie (see bug #3 below) so authenticated navigation doesn't pay a consent query on every request. Terms/Privacy content ships as versioned constants (`TERMS_VERSION`/`PRIVACY_VERSION`, `src/lib/legal/versions.ts`) at `/terms` and `/privacy`.

**Alternatives:** keep magic-link sign-in as a fallback alongside password (rejected — two parallel auth UIs to maintain and explain; the reset flow already covers the "I forgot" case a password-only design needs, and DECISIONS' prior magic-link-only choice was itself undocumented technical debt this slice retires); phone/SMS auth (deferred — needs a paid SMS provider, no product urgency yet); collecting a username at sign-up instead of only at onboarding (rejected — a username field at sign-up duplicates a form field onboarding already owns, and pre-onboarding accounts have nothing to disambiguate a username against yet).

**Reasoning:** this directly resolves the "re-authentication is email-only" gap the user flagged as real friction and asked to defer only temporarily (see the now-removed KNOWN_LIMITATIONS entry from 2026-07-17: "I'll deal with it for now, but this needs to change eventually"). Consent-as-append-only-rows (rather than a boolean flag) gives an audit trail that survives a future Terms/Privacy version bump — bumping `TERMS_VERSION` alone re-prompts every user through the same gate, with no migration needed. Three real bugs were found via live testing against the actual Supabase project during this slice — none caught by unit tests or code review alone — and fixed within the same run:

1. **Email enumeration via signup, found during live QA, fixed in the signup action.** `signUpWithPassword`'s original anti-enumeration design assumed Supabase always returns an obfuscated user object (no identities) for a duplicate-email signup. Live testing found this Supabase project has "Confirm email" disabled, so a duplicate signup for an already-CONFIRMED email actually returns a real `email_exists`/"already registered" *error*, not an obfuscated user — producing a visibly different response message than a genuine new signup and letting an attacker distinguish registered from unregistered emails by message text alone. Fixed with `isAlreadyRegisteredError()` (checks both error code and message text, since Supabase's exact error shape has varied across SDK versions), which now short-circuits to the identical "check your email" message regardless of which path Supabase took. Live-re-verified end-to-end against the real project: the same email submitted twice produced the identical message both times.
2. **Open-redirect hardening, found by an automated background security scanner after the callback route landed, not by the task reviewer.** `/auth/callback`'s `next`-param guard (`startsWith("/") && !startsWith("//")`) matches a pattern that's sometimes bypassable elsewhere via backslash-normalization tricks (some URL parsers treat `\` as `/`). Hardened with an added `!next.includes("\\")` check. The original guard was likely already safe in this route's specific code shape — `origin` is prepended as a literal string prefix before any URL parsing occurs, so there's no parser to fool — but the hardening removes any doubt at negligible cost.
3. **Consent-gate cookie not bound to user identity, self-caught during the consent-gate task's own implementation.** The `pfi_agreed` cookie cache was originally value-bound only to document versions, not to which user consented, and `signOut()` didn't clear it. On a shared browser: User A consents (cookie set) → signs out → User B signs in without ever consenting → the proxy's cookie check incorrectly short-circuits the DB check, letting User B into the app having never consented. Not a data leak (RLS still isolates each user's actual data correctly) but a real compliance/correctness gap in the one feature whose entire job is tracking consent. Fixed by binding the cookie value to the user id (`agreedCookieValue`: `` `${userId}:${TERMS_VERSION}|${PRIVACY_VERSION}` ``) and having `signOut()` delete the cookie explicitly as defense-in-depth (harmless once the value is user-bound, but stops it lingering into the next sign-in on the same browser regardless). A reviewer independently traced the fix by hand across the full shared-browser scenario and confirmed it closes the gap.

## 29. 2026-07-19 — Password-reset/signup links redeemed via token_hash, not PKCE code; production Site URL was still localhost

**Decision:** email-link redemption for password reset and signup confirmation moved to a new `/auth/confirm` route using `token_hash` + `supabase.auth.verifyOtp()` — Supabase's documented device-independent redemption mechanism — instead of `/auth/callback`'s `?code=` + `exchangeCodeForSession()`. `/auth/callback` is kept, unmodified in behavior, as the landing spot for any future `?code=`-shaped flow (e.g. an OAuth provider); nothing in this codebase generates a link pointing at it anymore once the Supabase dashboard's two affected email templates ("Reset Password", "Confirm signup") are switched to the new format. Separately, `supabase/config.toml`'s `site_url` — which the entire fix turned out to hinge on — was corrected from `http://localhost:3000` to `https://pfi-one.vercel.app`.

**How this was found:** live, on the real `pfi@aigaai.com` account, immediately after PR #16 shipped. A password-reset link redirected the user not to the password-reset screen but into an unrelated, already-logged-in account's onboarding flow, with no visible error. Root cause: PKCE's `code_verifier` cookie is set by whichever browser storage context calls `resetPasswordForEmail`; iOS keeps an installed PWA's storage separate from Safari, and an emailed link always opens in Safari regardless of where the request originated. The code exchange failed silently, `/auth/callback` fell through to `/login?error=link`, and because the browser happened to have a different account already authenticated, the proxy's already-signed-in redirect bounced the user into that account instead — the actual symptom reported ("it's making me recreate my company/username").

**Two additional bugs surfaced and were fixed before this was fully working, each caught by a step this project already practices:**
1. A first version of the fix repointed `signUpWithPassword`/`requestPasswordReset`'s `redirectTo`/`emailRedirectTo` away from `/auth/callback` in the same commit that added `/auth/confirm` — an independent review caught that this would have broken the *live* reset flow the moment it deployed, since the CURRENT (unchanged) email templates build their links from `{{ .RedirectTo }}` and nothing else in the codebase calls `exchangeCodeForSession`. Reverted to the exact deployed values (confirmed byte-identical via `git diff`), making the code change purely additive until the dashboard templates are manually switched.
2. The same review caught that the dashboard template's `next=` param, if built from `{{ .RedirectTo }}` as the code comments originally suggested, resolves to an *absolute* URL — which `safeRedirectPath()` correctly rejects (open-redirect guard), stranding an authenticated user on `/` instead of the intended page. The route's doc comment now specifies a hardcoded relative `next=` value, matching Supabase's own documented example, and this was live-verified against the real project (relative `next=` reaches the intended page with a session; an absolute one safely falls back to `/` without crashing).
3. After the dashboard templates and `/auth/confirm` code were both live, the *actual* first end-to-end attempt on the real account still failed at the "save new password" step with a generic error. Reproducing the exact `verifyOtp` → `updateUser` sequence against a fresh throwaway account succeeded immediately, and a second reproduction against the real `pfi@aigaai.com` account also succeeded (mutating its password as a side effect of diagnosis — a mistake; should have stopped at the first successful reproduction rather than repeating it against the real account). The working theory: a Vercel deployment was actively rolling out mid-attempt, and the browser's already-loaded page held a server-action reference from a different build than what was currently serving. The user's own next attempt, after the deployment settled, succeeded.

**Alternatives:** keep `/auth/callback` for reset/signup and just tell users to complete the flow in the same browser context they started it in (rejected — the whole point of shipping an installable PWA is that it's the primary interface; asking users to avoid it for a core flow isn't an acceptable answer); switch the Supabase client's global `flowType` to implicit (rejected — that would remove PKCE's stronger security properties for login and signup too, a real regression, to fix a problem `token_hash` already solves without touching those flows).

**Reasoning:** `token_hash` carries the credential in the URL itself and needs no pre-existing browser state to redeem, which is exactly the property this bug needed — confirmed by redeeming a real generated `token_hash` via a bare `fetch()` with zero cookies attached. Supabase does not sync email templates via `config push` (confirmed against their docs), so the dashboard edit is unavoidably a manual, human step — same category as Task 12, and for the same reason: it changes live auth behavior and requires dashboard access this session doesn't have.

**Consequences:** `/auth/callback` is now dead code from the app's own perspective (nothing links to it) but is kept, cheaply, as a landing spot for a future OAuth-provider flow. Because a single Supabase project is shared between local dev and production, `site_url` being set to production means a password reset requested from a local dev server now emails a link pointing at production, not localhost — this was already true in the other, broken direction before this fix, and local/e2e testing never depended on real email-link delivery to begin with (it uses `admin.generateLink`/`hashed_token` directly). `additional_redirect_urls` (the actual redirect allowlist, separate from `site_url`) was confirmed live-tested and unaffected — it already covered `/auth/confirm` via the existing `https://pfi-one.vercel.app/**` wildcard entry.

One plan-document correction was also made mid-implementation: the manual Supabase-dashboard checklist (Task 12, not yet executed — see below) originally asserted "Confirm email is already ON (it already is — magic links used it)." That was a false assumption — magic-link/OTP sign-in never required "Confirm email," and the project currently has it OFF, which is exactly what let bug #1 above happen. `docs/superpowers/plans/2026-07-19-password-auth.md` was corrected in place to say "Enable Confirm email" as an explicit required action, with a note on the residual gap this leaves open even after bug #1's fix: with Confirm-email OFF, a genuinely-new signup sets a session cookie immediately while an already-registered attempt does not — a response-*shape*/cookie-timing side channel independent of the message body, distinct from the message-level enumeration bug #1 closed, and only fully closed once Confirm email is enabled.

**Consequences:** `SUPABASE_SERVICE_ROLE_KEY` is now used at request time (username resolution, sign-up consent insert), not just in scripts/migrations — it was already a required production secret, but its blast radius if leaked has grown. Terms/Privacy are drafted, versioned copy pending legal review (KNOWN_LIMITATIONS). Existing/demo accounts that never had a password set it via the reset flow. **Task 12 (Supabase dashboard settings: enable "Confirm email," configure leaked-password protection, verify redirect URLs) is a manual checklist requiring dashboard access and has deliberately NOT been executed** — it changes live production auth behavior (sign-up would stop granting an immediate session, requiring an email-verification click instead) and is held for the user's explicit review rather than applied silently. Until it runs, the response-shape/cookie-timing side channel described above remains open (KNOWN_LIMITATIONS).

**Update (2026-07-19, later the same day — Task 12 executed and verified live).** The checklist above has now been run against production, and the side channel is closed. **Reasoning:** Generalizing the pipeline is a direct payoff of slice 1's original Zod-boundary design (DECISIONS #26) — the boundary was already the only channel between engine and model, so widening it to a discriminated union of inputs (rather than re-deriving a second boundary) keeps "deterministic code calculates; AI only narrates" enforced structurally for the new surface at close to zero marginal risk. Single-call-per-visible-set caching matches slice 1's cache-by-input-hash design (any change to the visible drivers naturally invalidates the row) and avoids the N-separate-calls cost/latency multiplier a per-card design would have paid on every dashboard load. Defensive per-card matching (rather than trusting driver array order/length to stay aligned across chart-range switches) was necessary because the AI input is always built over the fixed 30-day default window while the UI's visible drivers follow whatever chart range the user has selected — matching on type+date+rounded-impact, not array position alone, is what makes range-switching degrade per-card instead of either crashing or silently mismatching one event's AI text onto a different event's card.

**Consequences:** the "View transactions" drill-down link relocated from the card itself into the expanded panel (one tap deeper than before, but nothing lost — same URL/query params). `WhatMovedYourLine`'s `expandedId` state isn't cleared when the `drivers` prop changes (switching chart range can leave a stale expanded id that silently re-opens if the user switches back to a range containing that same event id) — tracked in KNOWN_LIMITATIONS, not fixed in this slice (out of scope for a testing/docs task). AI explanations only ever cover the fixed 30-day window the input was built over; every other range (90D/1Y/All) always shows deterministic "Calculated" text for every driver, by design, not as a bug. This slice's e2e coverage and live visual verification (Task 9) exercised only the keyless/deterministic path — `AI_GATEWAY_API_KEY` is forced empty in `playwright.config.ts`'s `webServer`, matching slice 1's precedent that a real-model pass is a separate pre-merge gate (Task 10), not something a keyless review can substitute for.

## 33. 2026-07-20 — `SCORE_METRIC_CONCEPT_IDS` is a label→term binding, distinct from `dataMetricKey`

**Decision:** `src/lib/concepts/score-term-map.ts`'s `SCORE_METRIC_CONCEPT_IDS` (a score-dimension metric id → `ConceptId` map, e.g. `net_cash_flow_margin` → `free-cash-flow`) is a deliberately separate concern from a `FinancialConcept`'s existing `dataMetricKey`. `dataMetricKey` answers "which engine field feeds this concept's lesson personalization" (a data binding, read by the Academy lesson layer). `SCORE_METRIC_CONCEPT_IDS` answers "which concept's definition does this displayed metric *label* teach" (a label-teaching binding, read by `FinancialTerm` when wiring score-dimension rows) — the score UI shows metric labels like "Free cash flow margin" and "Typical monthly free cash flow" that don't map 1:1 to a single concept's `dataMetricKey`, so a second, purpose-built map was added rather than overloading `dataMetricKey` to serve both jobs.

**Alternatives considered:** reuse `dataMetricKey` for score-row wiring too (rejected — a concept's `dataMetricKey` points at one engine field for lesson personalization; several score metrics with different ids and different labels legitimately teach the same concept, e.g. both `net_cash_flow_margin` and `recurring_surplus` map to `free-cash-flow`, which a single 1:1 field can't express); derive the concept id from the metric id string at render time via a naming convention (rejected — score metric ids and concept ids are independently owned vocabularies with no reliable mechanical mapping, e.g. `liquid_runway_months` → `liquidity`).

**Reasoning:** keeping the two maps separate keeps each one honest about what it answers, and keeps `FinancialConcept` records free of UI-wiring concerns (a concept doesn't need to know which score metrics happen to display its name). `score-term-map.test.ts` validates every mapped id resolves to a real, published concept.

**Consequences:** adding a new score metric that should be tappable requires a `SCORE_METRIC_CONCEPT_IDS` entry in addition to (not instead of) any `dataMetricKey` binding the underlying concept already has; the two maps can point at the same concept for different reasons and that's expected, not a duplication bug.

## 36. 2026-07-21 — CSV mapping is automatic-first with a privacy-scoped AI fallback

**Decision:** Standard bank CSVs bypass the mapping screen when deterministic header, date-order, account-aware sign, and row-shape checks are confident. The preview identifies that PFI mapped the file and provides a `Change mapping` escape hatch. Ambiguous files use a plain-language confirmation screen. An optional authenticated AI Gateway call may suggest unfamiliar column roles and bank category-label mappings, but receives structural column profiles rather than transaction values and never controls monetary parsing. ## 38. 2026-07-22 — Two dashboard numbers renamed for clarity: "PFI" (the index) and "Fundamentals Score"

**Decision:** Rename the two headline numbers so they read as different *kinds* of
thing. The indexed performance chart, formerly labeled **"Personal Index,"** becomes
**"PFI"** (the Personal Finance Index — a price/level that starts near 100, no ceiling,
moves daily like a share price under the `$TICKER`). The 0–900 health score, formerly
labeled **"PFI Score,"** becomes **"Fundamentals Score."** User-facing only: the card

churn with no user benefit and touches the score-versioning machinery.

**Alternatives:** merge into one number (rejected — the index-vs-score distinction is
the "household as a public company" thesis: a company has both a share price *and*
fundamental health, and they can diverge); rename the chart to "PFI Score" and the
0–900 to "Fundamentals Score" (the user's first proposal, rejected — it leaves *both*
numbers ending in "Score," reintroducing the sound-alike collision, and mislabels an
unbounded index as a bounded score); keep "Personal Index" for the chart (rejected — it
is an abbreviation of the product name "Personal Finance Index," which is exactly why it
read as a twin of "PFI Score"). Chose "PFI" over the full "Personal Finance Index" for
the chart card because the `$TICKER` above it already supplies the stock framing and the

problem — when the index drops on a one-day cash dip while the 90-day Fundamentals Score

added `.claude/**`, `test-results/**`, and `playwright-report/**` to the eslint
`globalIgnores` (`eslint.config.mjs`): the bare `eslint` `lint` script traversed the

sources, which had been silently failing `pnpm check`'s lint step.
