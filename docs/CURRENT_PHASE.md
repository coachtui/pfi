## Completed (this phase — Academy content-refinement Slice A: schema extension + reference implementations)

This slice refines the Academy's *content and presentation* rather than its
architecture: Slice 3 shipped routes, progress tracking, and the lesson/sheet
shells; lessons still read like generic financial documentation and
definition sheets gated their most useful content (why it matters, business
context) behind lesson completion, contradicting the product's
never-locked-comprehension rule. Slice A validates the new content
framework on two reference concepts — Revenue (a full lesson) and Available
Capital (a glossary-only, PFI-specific sheet) — plus the schema extension
and shared components both depend on, before Slices B/C propagate the
pattern to the other 9 lessons and 13 remaining definition sheets. Built
directly on top of the still-unmerged Academy Slice 3 on
`worktree-academy-content-refinement`, Tasks 1–9 of 9; DECISIONS #35.

- **Task 1 — classification + schema extension (`1c65e59`).** `FinancialConcept`
  gains a required `classification` field (`standard_finance` /
  `household_adaptation` / `pfi_metric`) plus optional
  `plainEnglishSummary`, `memorableDistinction`, `formulaRows`,
  `comparisonRows`, `interpretation`, and `whereUsed` fields. Registry
  validation added for `formulaRows` (requires the plain-text `formula`
  fallback) and `whereUsed` (rejects empty entries). All 15 existing
  concepts classified per the spec's audit table; `content.test.ts` asserts
  every concept is classified and registry rules pass.
- **Task 2 — lesson field renames with stable check ids (`822c781`).**
  Collapses three near-identical `KnowledgeCheck` union arms into one
  interface with a stable `id`; renames `Lesson.intro` → `opening` and
  `knowledgeCheck` → `knowledgeChecks`; threads `checkId` (replacing the
  positional `checkIndex`) through the registry validator, progress
  derivation, `answerKnowledgeCheck`, and all three lesson React
  components. Applied mechanically across all 10 lesson-bearing content
  files with `<concept-id>-check-<n>` ids — no prose changes. Changed
  pre-launch, while `academy_progress` holds no real user data, since a
  positional index breaks silently the moment checks are reordered or
  edited.
- **Task 3 — shared presentational components (`dcaaeda`).**
  `ClassificationLabel`, `FormulaBlock` (structured rows with a plain-text
  fallback), `ComparisonRows` (included/excluded rows for a memorable
  distinction), and `WhereUsedList` — pure presentational components shared
  by both the definition sheet and the lesson page so the two surfaces
  render identically without duplicating markup.
- **Task 4 — concept live-data resolver (`2e0747f`).** A pure
  `computeReportLive` resolver turns a concept's `metricKey` (e.g.
  `"report:revenue"`) into display-ready current/prior figures and a delta;
  a thin Supabase-backed `getConceptLiveData` wrapper and the
  `getConceptLive` server action feed the definition sheet's lazy
  completed-state fetch. `queries.ts`'s top-level `"server-only"` import (it
  throws under Vitest's plain Node environment as soon as it's loaded, even
  transitively) is deferred to a dynamic import inside
  `getConceptLiveData` so the pure resolver stays unit-testable in
  isolation, mirroring the existing pure/impure split elsewhere in this
  directory.
- **Task 5 — progress-aware, un-gated definition sheet (`b333650`).**
  Reverses Slice 3's completion-gating: `whyItMatters`, `classification`,
  and `businessContext` now render at every progress state, not only after
  completion — comprehension is never locked, matching the product's
  binding rule. Adds a three-state CTA (Take/Continue/Review the lesson)
  and a completed-only live-data block backed by Task 4's resolver.
  `getCompletedConceptIds` (a boolean signal) becomes `getAcademyStatusIds`
  (in-progress + completed sets) to drive the new three-state CTA.
- **Task 6 — lesson framework upgrade (`42bea65`).** The lesson template
  gains a memorable-distinction callout with included/excluded comparison
  rows (right after the opening section), a statement-style visual
  calculation (`FormulaBlock` with real numbers, "Sample figures" pill), a
  household-application section reading real household data server-side
  when available ("Calculated from your data") and falling back to
  `genericExample` otherwise ("Sample household"), and a completion card
  rewritten in fluency-framed language (no "unlocked"/"analytical depth"
  copy) with Review concept / Back to Academy / Next actions.
- **Task 7 — Revenue reference lesson (`d795ac3`).** `revenue.ts` fully
  rewritten against the new schema: the memorable distinction ("Not every
  deposit is revenue.") with five comparison rows, structured
  `formulaRows`, an `interpretation` section, and a `whereUsed` list —
  content-only, no framework changes.
- **Task 8 — Available Capital reference sheet (`04c8ad3`).**
  `available-capital.ts` rewritten as the glossary-only, `pfi_metric`
  reference implementation: classification, a visual `FormulaBlock`, and
  `whereUsed`. Found and removed one real internal-language leak in this
  concept's existing copy — an "audit ruling, spec finding #6" phrase that
  rendered verbatim in the live sheet — guarded going forward by a new
  content test asserting no concept's rendered strings match internal
  process vocabulary (`/audit ruling/i`, `/spec finding/i`, `/\btask \d/i`,
  `/decisions #/i`, `/implementation plan/i`).
- **Task 9 (this task) — e2e updates, docs, full check, live
  verification.** `e2e/academy.spec.ts` updated in place (not rewritten):
  the Revenue deep-link test gained un-gating assertions (classification

  covers the in-progress "Continue lesson" CTA; the completion test asserts
  the "Review concept" button; the completed-sheet test is renamed ("the
  completed term sheet deepens with the user's data") and now asserts
  "Academy concept completed" and the "Your data" live block instead of the

Slice 3 of the Academy MVP ships the part users actually spend time in: a
home screen showing real progress, a full lesson experience with knowledge
checks that teach rather than gate, and DB-backed progress persisted behind
owner-only RLS — status always derived, never stored. It also unlocks the
term sheet's "depth" fields (`whyItMatters`/`businessContext`) once a lesson
is complete, closing the loop Slice 2 left open. Built on
`worktree-academy-slice3`, Tasks 1–10 of 10; DECISIONS #34; **not yet
merged — pending final whole-branch review.**

- **Task 1 — `academy_progress` table + RLS (`18e85ef`).** New migration
  `0012_academy_progress.sql`: `(user_id, concept_id)` composite primary key,
  `started_at`/`completed_at`/`check_responses` (a raw `{ checkIndex,
  choiceIndex }[]` jsonb array — correctness is never persisted, it's
  derivable from the compile-time registry), owner-only RLS (select/insert/
  update/delete), no FK on `concept_id` since concepts live in code, and no
  ownership trigger needed (the only FK is `user_id`, same reasoning as
  `ai_narrations`). RLS suite grew by 6 checks (owner insert/read, cross-user
  read/insert-forgery/update/delete denial).

  `src/lib/concepts/progress.ts`: `conceptStatus` (not-started/in-progress/
  completed from a nullable `ProgressRow`), `academyTallies` (lessons/modules
  completed + percent, registry-driven so the total can never drift from the
  actual concept count), `nextUpLesson` (first incomplete lesson in module
  order), `recentlyCompleted`, `appendCheckResponse` (first-answer-wins,
  flags `allAnswered` for the completion trigger), and `validateCheckAnswer`
  (bounds-checks a check/choice index against the real registry before any
  write). No React/Next imports, same extraction discipline as
  `financial-engine`; fully unit-tested (`progress.test.ts`).
- **Task 3 — term-sheet completed variant (`bfee198`).**
  `buildTermSheetModel` gained `hasLesson`/`completed` fields — `completed`
  is passed in by the caller (the layout-level completed-ids fetch, Task 8),
  never looked up by the pure function itself — and `TermSheetModel`'s
  `whyItMatters`/`businessContext` are now only populated once `completed` is
  true, keeping the pre-completion sheet exactly as spare as Slice 2 shipped
  it.
- **Task 4 — progress queries + server actions (`550103d`).**
  `src/lib/data/queries.ts` gained `getAcademyProgress`/
  `getCompletedConceptIds`; `src/app/actions/academy.ts` gained

- **Task 5 — Academy bottom-nav tab (`a2f835f`).** `BottomNav.tsx` gained a

  `role="status"` "Lesson complete" block with a "Back to Academy" link the
  moment the last check is answered; `LessonView` wraps both in a
  keyboard-navigable (Arrow-key) Lesson/Related `role="tablist"` and a
